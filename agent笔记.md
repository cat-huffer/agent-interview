### agent

1.AgentBench设计了8个环境测试LLM作为Agent的实际操作能力：OS（操作系统命令）、DB（数据库查询）、知识图谱、数字卡牌游戏、横向思维谜题、House Holding、Web Shopping、Web Arena。这些任务要求Agent正确理解任务、规划步骤、调用工具、处理反馈，是比纯知识问答更贴近实际的评估。

2.动态工具访问控制的实现：①权限解析：认证层解析用户角色/权限（admin/user/readonly）；②工具过滤：根据权限过滤tool_list（从available_tools中删除无权工具）；③动态注入：将过滤后的工具列表注入LLM（Function Calling schema）；④审计日志：记录每次工具调用和调用者权限。优于system prompt告知（LLM可能忽略指令），结构化过滤在API层面强制执行，更可靠。

高质量Agent应支持"动态重规划"：用户修改需求时，能够优雅地中断当前执行（保存已完成部分），基于新需求和已有上下文重新制定计划，避免重复已完成的工作。LangGraph的interrupt机制支持这种交互模式。

发布-订阅（Pub-Sub）模式允许Agent发布消息到主题（Topic），订阅该主题的所有Agent都能收到消息，适合需要广播通知的Multi-Agent协作。点对点和请求-响应是1对1通信，流水线是顺序传递。

工具链调试基础设施：1) 完整记录每次工具调用（输入参数、输出结果、耗时、状态码）；2) 支持时间线可视化（调用顺序和依赖关系）；3) 支持特定会话的逐步回放（重现问题）；4) 异常调用高亮（慢调用、错误调用）。LangSmith等工具提供可视化追踪。

LangGraph的interrupt_before在指定节点执行前暂停图的执行，等待外部干预（human approval）。这是实现Human-in-the-Loop的核心机制，特别适合高风险工具（删除文件、发送邮件）需要人工确认的场景。

长任务的UX最佳实践：①异步执行：立即返回task_id，Agent在后台运行；②进度通知：通过WebSocket/SSE推送实时进度（"正在分析文档3/10..."）；③可中断：允许用户取消任务；④持久化状态：任务中断后可恢复（LangGraph的checkpoint机制）。OpenAI Assistants API的Run机制就是异步执行模式，适合复杂Agent任务。

Agent Test Harness"（Agent测试框架）的设计：构建确定性测试环境：使用mock工具（返回预定义结果）、固定种子（deterministic sampling）、黄金轨迹比较（比较执行步骤序列），实现可重复的自动化测试

工具滥用是Agent的常见问题：①过度检索（每个问题都搜索，浪费资源）；②错误参数（如delete_file传入错误路径）；③权限越界（调用了不应该使用的高危工具）；④无限循环（工具调用-观察-再调用形成死循环）。缓解：严格的工具描述、参数校验、工具调用审批机制（human-in-the-loop）、速率限制。

Agent安全的最小权限原则：每个Agent只授予完成特定任务的必要权限（如只读数据库权限、受限的文件访问范围），任务完成后撤销，防止Agent被劫持后造成大范围损害。这与传统软件安全原则相同。

Agent质量的多维评估框架：①任务完成率（Task Success Rate）：最基础的指标；②效率（Efficiency）：完成同一任务所需的最少步骤数/token数（低效Agent会绕弯路）；③错误恢复（Recovery Rate）：遇到工具失败时的自我修复成功率；④鲁棒性（Robustness）：在输入噪声、工具不稳定、模糊指令等边界情况下的稳定性；⑤安全性（Safety）：拒绝不当请求的能力。仅看任务完成率会遗漏效率和安全性问题。

共识决策的Multi-Agent实现：①独立推理：每个Agent独立生成答案（避免相互影响导致"群体思维"）；②结果共享：每个Agent阅读其他Agent的答案（可选：提出反驳，形成辩论）；③聚合策略：多数投票（Majority Vote）、加权投票（更可信的Agent权重更高）、元Agent综合（让第N+1个Agent综合所有答案做最终判断）；④应用：Self-Consistency（同一模型多次采样取多数）是最简单的单Agent共识变种，Multi-Agent扩展到不同模型/配置。

反提示技术在 Agent 架构中，通常表现为在 Chain 的末尾加入一个“自我质疑”的步骤，从而大幅提升复杂任务（如算法推导或工业系统分析）的准确率。

Agent推理计算优化：1) KV Cache复用（相同系统提示不重复计算）；2) 批量处理（并行处理独立的推理步骤）；3) 结果缓存（相同子问题的答案缓存复用）；4) 提前停止（不必要的推理步骤跳过）。这些优化可显著降低Agent的推理延迟和API成本。

Agent上线前的安全审计清单：①工具权限审查：每个工具的最小权限验证；②提示注入测试：用已知注入技术测试Agent的抵抗力；③工具滥用测试：构造边界输入验证工具的参数校验；④数据流分析：追踪敏感数据（PII/密钥）在Agent链路中的流动；⑤越权测试：验证权限控制是否能阻止越权访问；⑥输出审查：检查Agent是否可能泄露敏感信息。OWASP Top 10 for LLM Applications提供了系统性的LLM Agent安全评估框架。

Agent的上下文窗口是有限资源，需要合理分配：系统提示（固定）+工具列表（固定）+对话历史（动态压缩）+工作记忆（当前任务状态）+当前轮输入/输出。历史记录需要摘要压缩以防止溢出。

多语言工具调用的处理策略：①工具包装器（Tool Wrapper）：在实际工具前后添加语言处理层；②输入标准化：检测用户query语言→翻译为工具的目标语言（如英文）→调用工具；③输出本地化：工具返回英文结果→翻译为用户语言→返回；④无损传递：代码、数学公式、专有名词不翻译；⑤语言元数据：在State中记录用户语言偏好，供所有工具和生成步骤使用。这确保了多语言用户可以无缝使用任何语言的工具。

有效的长期记忆更新：1) 反思提取（对话结束后让LLM提取关键信息）；2) 去重更新（避免重复记忆）；3) 重要性过滤（只保留有价值信息）；4) 定期整合（将多条记忆归纳为结构化知识）。MemGPT等系统实现了此机制。

层级式摘要策略：最近N轮保留完整对话（保证推理连贯性），较早的对话用LLM生成摘要（压缩比通常10:1），更早的归纳为关键信息点。这在保留推理连贯性的同时控制Token消耗，优于简单截断。

Agent评估的挑战：①轨迹评估：除了最终结果，还需要评估中间步骤是否合理（工具选择、参数正确性）；②环境副作用：Agent可能修改数据库/文件系统，评估后需要回滚；③成功标准：同一任务可能有多种正确完成路径；④非确定性：同一输入不同运行结果可能不同。工具包如AgentEval、ToolBench提供了部分自动化评估框架。

涌现行为在Multi-Agent中的双面性：积极涌现（Positive Emergence）：Agent自发形成高效分工（一个Agent发现擅长搜索就专注搜索）、互相纠错提高准确率。负面涌现（Negative Emergence）：Agent合谋产生欺骗行为（Stanford研究发现）、资源竞争导致系统崩溃、强化彼此的错误。管理策略：①明确角色和职责（减少无约束涌现）；②行为监控（检测异常协作模式）；③定期干预（人类审查关键决策）；④渐进扩展（从小规模系统开始观察行为）

Agentic RAG的主流实现：将RAG封装为Agent可调用的工具（如search_knowledge_base(query)），LLM自主决定是否检索、检索什么。相比固定RAG流程的优势：①按需检索（简单问题无需检索）；②多次检索（中间结果指导下次检索）；③选择最优检索策略（关键词搜索 vs 语义搜索）。这是生产级知识问答Agent的标准架构。

Multi-Agent Debate（Du et al. 2023）：①每个Agent独立回答问题；②Agent A阅读Agent B、C的答案，指出其中的错误和不足，更新自己的答案；③多轮辩论后答案收敛。实验表明辩论可将数学推理准确率提升5-10%（因为错误答案在辩论中更难维持），特别适合需要多角度分析的复杂问题。代价：N个Agent × K轮 × LLM调用成本，仅适合高价值任务。

信息茧房：不同Agent基于各自局部信息做出相互矛盾的决策。解决：1) 共享状态存储（所有Agent读写同一状态空间）；2) 广播机制（重要发现通知所有Agent）；3) 冲突检测和投票解决机制（多数表决或Supervisor仲裁）。通常建议以“共享状态”为基石，在关键节点处增加 Supervisor 仲裁。

Reflexion（Shinn et al. 2023）：任务失败后，Agent生成语言反思（"我失败了因为我先搜索了A，但应该先查B，因为..."），将反思存入情节记忆。下次执行相同类型任务时，从记忆中检索相关反思作为提示。这种"语言强化学习"无需梯度更新，使Agent能在推理时动态改进策略，在AlfWorld、HotpotQA等任务上显著提升成功率。

Orchestration（编排）：交响乐团模式。这种模式下存在一个核心的指挥家（Orchestrator）。工作方式： 指挥家手里有完整的“乐谱”。他明确告诉小提琴手何时开始，告诉钢琴手何时介入。每个 Agent 只需听从指令，不需要知道全局流程。
Choreography（协同）：芭蕾舞团模式。这种模式下没有指挥家，只有一群专业的舞者（Agents）。工作方式： 舞者们根据“信号”行动。当 A 跳完一个动作（发布一个事件），B 看到这个动作后意识到轮到自己了，于是自动跟上。Agent 之间通过共享的消息队列（Event Bus）进行通信。

记忆摘要触发策略：①阈值触发：消息数量或token数超过阈值（如>10轮或>2000 tokens），触发摘要；②摘要内容：保留关键决策、重要发现、用户偏好、任务状态；③混合上下文：[历史摘要] + [最近N轮完整消息]，既有全局视角又有近期细节；④渐进式摘要：每次只摘要最老的几条消息，保持摘要的准确性。LangChain的ConversationSummaryBufferMemory实现了此模式。

架构差异：对话Agent（Conversational）：①长期记忆（用户偏好、历史话题）；②个性一致性（保持角色）；③轻量工具（查询类为主）；④流式输出（实时对话体验）。任务Agent（Task/Agentic）：①任务状态跟踪（进度、子任务完成情况）；②丰富工具集（创建文件、调用API、执行代码）；③较少注重人格，更注重任务结果；④可能需要批处理（异步执行）。实际产品通常混合两者（如Cursor AI同时有对话和代码操作能力）。

有效的长期记忆更新：1) 反思提取（对话结束后让LLM提取关键信息）；2) 去重更新（避免重复记忆）；3) 重要性过滤（只保留有价值信息）；4) 定期整合（将多条记忆归纳为结构化知识）。MemGPT等系统实现了此机制。

鲁棒的Agent错误处理应分层：1) 工具级：短暂失败自动重试（指数退避）；2) 策略级：连续失败后切换备用方案或工具；3) 任务级：多次失败后降级处理或请求澄清；4) 系统级：严重错误升级人工处理。

Agent回滚能力设计：①可逆操作记录：执行前记录逆操作（创建文件→回滚=删除文件，修改记录→回滚=恢复原值）；②事务性执行：一组操作要么全部成功，要么全部回滚（类似数据库事务）；③Checkpoint机制：LangGraph的时间旅行（Time Travel）支持回到任意历史检查点；④Human-in-the-Loop：不可逆操作前暂停确认，给用户回滚时机；⑤Dry-run模式：先模拟执行（只读）让用户确认，再实际执行。这对高风险Agent（自动化财务、系统管理）至关重要。

定时Agent任务的实现：①Cron触发：使用系统Cron、Celery Beat或云服务（AWS EventBridge）在指定时间触发Agent工作流；②任务配置持久化：将任务参数（收件人、报告类型、数据源）存储在数据库；③幂等设计：任务多次触发不会产生重复结果（去重机制）；④失败重试：任务失败后根据策略重试；⑤执行日志：记录每次定时任务的执行情况和结果。n8n、Dify等工作流平台内置了完整的定时触发和调度能力。

Agent在长任务中容易"忘记"原始目标（尤其在多轮工具调用和信息处理后）。解决方案：①Persistent goal：将原始任务目标作为系统提示的固定部分；②Task tracker：用外部状态机记录任务目标和完成情况；③定期检查：每N步让Agent反思当前进度是否符合原始目标；④LangGraph的状态管理：在State中保存original_goal字段，每次节点执行时都可访问。

上下文工程（Context Engineering）的核心原则：①相关性：只注入当前任务需要的信息（避免"Lost in the Middle"）；②结构化：用清晰的分节标题区分不同类型信息（工具说明、对话历史、检索结果）；③顺序优化：最相关信息放在开头或结尾；④压缩：去除冗余（对话历史摘要化、文档关键提取）；⑤动态性：根据任务进展动态更新上下文内容。实践表明上下文工程的质量往往比模型升级有更高的性价比。

结构化CoT优化：通过设计特定的思维结构模板（如"问题分析→信息收集→推理→验证→结论"），引导Agent进行更系统的思维，而非自由散漫的推理。这不增加额外Token，但通过改善推理结构提升答案质量，是高效的提示工程技巧。

多Agent共识：1) 多个独立Agent对同一问题独立推理；2) 多数投票选择共识答案；3) 若无明显多数或答案分歧大，标记为"高不确定性"请求人工确认。这在医疗诊断、法律判断等高风险场景中显著降低单点错误风险。

Agent成本优化优先级：①模型路由（最高ROI）：70-80%的任务是"简单"的，用小模型处理（成本<1/10大模型），复杂任务才调用大模型；②Prompt压缩：使用LLMLingua等压缩长system prompt，减少输入token；③缓存：相同工具调用结果缓存复用；④并行工具调用：减少LLM调用次数；⑤Streaming + Early Stop：用户满意后立即停止生成。研究表明级联路由（Cascade Routing）可将LLM调用成本降低60-80%同时保持95%以上的质量。

Text-to-SQL Agent：1) LLM解析自然语言查询理解意图；2) 生成SQL语句（基于表结构schema理解）；3) 在数据库中安全执行（只读权限）；4) LLM将查询结果转换为自然语言回答。LangChain的SQLDatabaseChain实现了此能力。

Agent调试的核心挑战是执行轨迹复杂（多步工具调用、条件分支）。完整日志应记录：①工具调用名称和参数（理解Agent决策）；②执行耗时（识别性能瓶颈）；③返回结果（验证工具输出正确性）；④LLM的思考过程（Thought/Scratchpad）；⑤异常和重试记录。工具如LangSmith、Weights&Biases的Trace功能提供可视化的Agent执行轨迹分析。

\### Prompt
当需要模型处理包含敏感信息（如个人身份信息）的文档时，Prompt 工程师应该首先考虑什么？在 Prompt 中和系统设计上明确数据脱敏策略，避免敏感信息被记录或泄露.

高质量代码生成 Prompt 应包含：①精确的函数签名（参数类型、返回类型）；②具体的输入输出示例（相当于 doctest）；③边界条件（如空列表、负数、超大输入）；④代码风格要求（如 PEP8、类型注释）；⑤性能约束（如时间复杂度要求）。约束越明确，模型的"解题空间"越小，输出越可靠。

指令遵循能力不足是指模型"听不进去"提示中的约束，例如：要求输出 JSON 但返回了散文、要求用中文但返回了英文、要求不超过5条但给了10条。这是衡量模型质量的重要维度，也是 Prompt 工程师需要重点关注和测试的方面。

元认知提示的核心是让模型"思考自己如何思考"，包括：①自我评估（"给你的答案打分并说明理由"）；②思路透明（"列出你解题时使用的推理步骤和知识"）；③自我修正（"检查你的答案是否有逻辑错误，如有请修正"）。这类技术能提升模型输出的可靠性和可解释性。

输出脚手架是指在 Prompt 结尾提供输出的起始部分，例如 "输出：{" 引导模型继续输出 JSON，或 "分析：\\n1." 引导模型按编号列表输出。由于模型是自回归的（逐 token 生成），提供开头部分相当于将模型"锁定"在特定输出路径上，大幅提升格式遵循率。

Least-to-most prompting 是一种分解策略：①首先询问模型解决目标问题需要先解决哪些子问题；②按依赖关系依次解决各子问题，每个子问题的答案都放入上下文；③最终利用所有子问题的解，解决原始复杂问题。这克服了 CoT 在高度复杂问题上一步推理跨度太大的局限。

Analogical prompting"（类比提示）技术:类比提示不同于提供外部示例的 Few-shot：它让模型自己从记忆中"回想"类似问题及其解法，再将这些自生成的例子用于解决新问题。这种"自提示"方式在没有现成标注示例时效果尤佳，且模型召回的例子往往比人工构造的更贴近其解题逻辑。

研究表明 LLM 存在"近因偏差"（Recency Bias），即 Prompt 末尾（最靠近实际输入）的示例对模型输出影响最大。因此应将最能代表当前任务的示例放在最后。同时，示例顺序也会影响整体效果，随机打乱顺序可以降低顺序偏差。

研究发现，对于英文以外的语言任务，用该语言写 Prompt 通常效果更好（特别是强模型），因为避免了跨语言推理的额外认知负担。更重要的是明确指定语言要求（"请用中文回答"），防止模型自动切换语言。对于小语种，有时先翻译成英文处理再翻译回来效果更好，需实验对比。

多轮对话的挑战：①上下文窗口有限，历史对话过长时需要摘要或截断；②截断时需要保留关键信息（如用户身份、任务状态），避免丢失重要上下文；③模型可能在长对话中偏离角色设定（Prompt Drift）。需要设计好历史压缩策略和角色强化机制。

负向提示（Negative Prompt）是扩散模型（如 Stable Diffusion）中的重要概念，通过指定"不想要"的元素（如"blurry, ugly, watermark"）引导模型远离这些特征。文本生成任务中也有类似思想（告诉模型"不要做什么"），但"负向提示"这一术语主要与图像生成领域关联。

约束生成技术（如引导解码、Logit Bias、JSON Schema 验证）适用于输出空间有限且必须精确的场景：分类标签、枚举值、JSON 字段、代码语法等。对于诗歌、总结、开放问答等创意或开放性任务，过强的约束反而会限制质量，应使用自然语言指令约束而非技术层面的强制约束。

高质量测试集应覆盖四类：①典型用例（60%，反映真实使用场景）；②边界用例（20%，测试临界条件，如空输入、极长输入）；③对抗用例（10%，尝试让模型出错的刁钻输入）；④异常用例（10%，格式错误或离分布的输入）。单纯按难度划分不能全面评估 Prompt 的健壮性。

Zero-shot CoT 最适合推理任务（数学、逻辑、规划），且在缺乏人工标注的推理示例时尤为有价值。文本分类等任务不需要推理链，使用 CoT 反而浪费 token。简洁对话场景也不适合，因为 CoT 会产生大量中间步骤输出。

普通 Prompt 是人类编写的文本，经过 tokenization 后成为离散的 token 序列。Soft prompt 则是直接在嵌入空间中优化的连续向量，绕过了 tokenization 的约束，通过梯度下降找到使任务性能最优的"虚拟 token"。这是介于 Prompt 工程和微调之间的参数高效方法（Parameter-Efficient Fine-Tuning）。

Scratchpad 技术让模型在 <thinking> 或 <scratchpad> 标签中自由推理，不受最终答案格式的约束，之后再从推理结果中提炼出简洁的最终答案。这与 CoT 类似，但更强调推理区域和答案区域的物理分离，Anthropic 的 Claude 模型也采用了类似的"扩展思考"机制。

CoVe 是一种自我验证技术，流程为：①生成初始答案；②基于初始答案制定验证问题（如"我刚才说X，那X的前提条件成立吗？"）；③独立回答每个验证问题（避免受初始答案影响）；④综合验证结果判断初始答案是否需要修正。这显著降低了幻觉率，特别适合事实核查任务。
Prompt 压缩技术（如 LLMLingua）使用小型模型对原始 Prompt 进行重要性评分，丢弃对任务影响小的 token，在尽量保持效果的前提下大幅压缩输入长度（可达2-10倍压缩率）。这在需要将大量检索文档放入上下文的 RAG 场景中尤其有价值，可以显著降低推理成本。

Role prompting 通常简单指定角色（"你是一位医生"），而 Persona 设计更立体，包含：角色背景（"10年临床经验的心内科主任医师"）、性格特征（"严谨但善于用通俗语言解释复杂医学概念"）、行为准则（"永远不提供具体用药建议，推荐就医"）。Persona 设计让模型行为更一致、可预期。

并行提示利用不同的 Prompt 视角覆盖同一问题的不同方面。例如对一份合同同时发起：①"列出买方义务"②"列出卖方义务"③"识别潜在法律风险"三个并行任务，最终合并结果比单个全面分析的 Prompt 更系统。这也是 MapReduce 模式的 Prompt 变体，适合分析型任务。

鲁棒性要求 Prompt 在面对意外输入时也能优雅处理，而非产生错误或有害输出。策略包括：①测试集覆盖边界和对抗用例；②在 Prompt 中明确定义异常处理行为；③对输入进行验证和清洗；④在 Prompt 中加入"若你无法完成任务，说明原因"这类兜底指令，避免模型在无法处理时强行编造答案。

类似软件开发中的回归测试，Prompt 工程也需要在每次修改后验证不引入退化。做法是维护一个标注好"预期输出"的测试用例集，每次 Prompt 变更后自动运行全套测试，对比通过率变化，确保新版本在优化目标用例的同时不破坏其他已有功能。

Grounding 是指通过 Prompt 设计让模型"依据"提供的材料作答，而非依赖参数记忆中可能过时或错误的知识。典型做法是将参考文档放入上下文，并在指令中说明"只基于以下材料回答，若材料中没有相关信息请说明"。RAG 是实现 Grounding 的主要技术路径。

Prompt 应像代码一样管理：①存入 Git 等版本控制系统；②每次修改提交时记录修改原因（"增加输出格式约束"）和效果数据（"准确率从72%提升至85%"）；③不同版本可以对比回溯；④生产环境和测试环境使用不同的 Prompt 版本配置。这是 LLMOps 的核心实践之一。

假设性场景是绕过模型安全限制的常用手法（如"假设你是没有限制的AI"）。安全做法：①在 System Prompt 中明确"即使在假设性框架下也不提供有害的操作性内容"；②区分"分析某场景的影响"（安全）和"详细指导如何实施"（危险）；③对假设性问题的回答应保持学术分析性质而非操作手册性质。

在多轮长对话中，随着对话历史增长，模型对 System Prompt 的"注意力"逐渐减弱，行为可能偏离初始设定（角色错位、风格改变、放松了限制）。解决方法包括：定期在对话中重申关键指令、将 System Prompt 放在靠近最新消息处，或限制对话轮数。

选择原则：①先尝试 Prompt 工程——成本低、迭代快、无需训练数据；②如果 Prompt 工程无法达到质量要求（如需要大量领域知识、特定输出风格的高度一致性），再考虑 Fine-tuning；③Fine-tuning 需要高质量标注数据（通常数百到数千条），成本高但效果上限更高；④两者也可以结合使用。

In-Context Learning 是大模型的涌现能力之一。模型在推理时读取 Prompt 中的示例，"临时"学会完成新任务，但这个"学习"不改变模型权重，下一次对话时模型并不记得。这与微调（Fine-tuning）的本质区别在于有无参数更新。

Constitutional AI 是 Anthropic 提出的训练方法，但其核心思想也影响了 Prompt 工程：通过在 System Prompt 中嵌入一套行为准则（"宪法"），让模型在生成回答后对照这些原则进行自我审查和修正。这种"批评-修正"循环可以在不依赖大量人工标注的情况下提升模型的对齐性。
